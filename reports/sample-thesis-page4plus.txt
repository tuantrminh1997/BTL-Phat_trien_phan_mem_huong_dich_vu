     II. GIỚI THIỆU CHUNG

      2.1 Mở đầu
      - Trong môi trường học tập, việc quản lý thông tin sinh viên (mã số, họ tên, giới tính,
ngày sinh, lớp) là một nhu cầu rất cơ bản. Đề tài này này hướng dẫn cách xây dựng một dịch
vụ web RESTful đơn giản để:
      + Cung cấp dữ liệu sinh viên,
      + Cho phép tìm kiếm/truy vấn theo yêu cầu,
      + Và dễ dàng mở rộng khi cần.
      - Trong phạm vi bài học, dữ liệu ban đầu được lưu trữ trong file sinhvien.xml để đơn
giản hoá việc thao tác (phù hợp cho tiểu luận, bài tập hoặc ứng dụng nhỏ). Về sau, nếu triển
khai trong thực tế, có thể thay thế bằng cơ sở dữ liệu quan hệ.
      - Mục tiêu của bài tập:
      + Xây dựng và thiết kế RESTful API với cú pháp rõ ràng, dễ sử dụng, đảm bảo mỗi tài
nguyên (resource) được định nghĩa và truy cập hợp lý.
      + Làm quen và thực hành với các công nghệ nền tảng quan trọng như JAX-RS, DOM
Parser, JSON Provider trong môi trường Java EE/Jakarta EE.
      + Triển khai trực tiếp trên NetBeans IDE kết hợp GlassFish Server để rèn kỹ năng phát
triển, triển khai và vận hành dịch vụ web.
      + Đóng vai trò client để gọi, kiểm tra và đánh giá API, qua đó hiểu trọn vẹn cả hai phía:
xây dựng dịch vụ và sử dụng dịch vụ.
      2.2 Các khái niệm và công nghệ nền tảng của RESTful Web Service
      2.2.1 REST
      - REST (Representational State Transfer) là một kiểu kiến trúc phần mềm cho dịch vụ
web, tập trung vào resource (tài nguyên). Mỗi tài nguyên (ví dụ: sinh viên, lớp học, môn học)
sẽ được định danh duy nhất bằng một URI.
      - Nguyên tắc chính:
      + Stateless (phi trạng thái): Mỗi yêu cầu (request) từ client (ứng dụng hoặc người dùng
gửi yêu cầu đến dịch vụ, ví dụ: trình duyệt web, Postman, ứng dụng di động) phải chứa đầy đủ
thông tin để server (máy chủ cung cấp và xử lý dịch vụ, nơi triển khai API hoặc ứng dụng
web) có thể hiểu và xử lý. Server không lưu lại trạng thái phiên làm việc trước đó của client.
      + Uniform Interface (giao diện thống nhất): Tất cả resource đều được truy cập theo một
cách giống nhau qua HTTP (giao thức truyền tải siêu văn bản) và địa chỉ URI (địa chỉ duy
nhất định danh cho từng tài nguyên).
      - Sử dụng các phương thức trong giao thức HTTP:
      5 | Page

      + GET: Lấy thông tin tài nguyên.
      + POST: Thêm mới tài nguyên.
      + PUT: Cập nhật tài nguyên.
      + DELETE: Xoá tài nguyên.
      - Representation (biểu diễn dữ liệu): Một resource có thể được trả về dưới nhiều dạng
khác nhau, phổ biến là JSON (JavaScript Object Notation: gọn nhẹ, dễ đọc, thường dùng trong
các ứng dụng web và mobile.) hoặc XML (eXtensible Markup Language: cấu trúc dạng thẻ, dễ
mở rộng, thường dùng trong các hệ thống cũ hoặc tích hợp với SOAP).
      Ví dụ:

      2.2.2 API REST
      API REST (RESTful API) là một giao diện lập trình ứng dụng (Application
Programming Interface) được xây dựng dựa trên nguyên tắc REST

      - API là "cầu nối" cho phép các phần mềm, ứng dụng khác nhau nói chuyện với nhau.
      - REST là một kiến trúc thiết kế dùng HTTP (giao thức web) để trao đổi dữ liệu.

      2.2.3 Công nghệ và công cụ sử dụng trong dự án (project)
      1. Java EE / Jakarta EE (GlassFish Application Server)
      - Là nền tảng để xây dựng và chạy ứng dụng web Java.
      - Trong project này, sử dụng GlassFish Server 5.1 làm application server (máy chủ ứng
dụng, nơi chạy và quản lý các ứng dụng web). Nó hoạt động như một container (môi trường
chứa và cung cấp các dịch vụ nền tảng cho ứng dụng, ví dụ: Servlet, JSP, JAX-RS) để triển
khai và vận hành dịch vụ web
      - GlassFish hỗ trợ đầy đủ các API chuẩn như JAX-RS, JAX-WS, Servlet, JSP, EJB.
      2. JAX-RS (Java API for RESTful Web Services)
      - Là chuẩn trong Java EE/Jakarta EE dùng để xây dựng các dịch vụ web theo mô hình
RESTful.
      - Jersey là thư viện triển khai (implementation) phổ biến nhất, và đã được tích hợp sẵn
trong GlassFish.
      - Khi lập trình với JAX-RS, ta chỉ cần dùng các ghi chú đặc biệt (annotation) để định
nghĩa REST API:
      @Path → Chỉ ra đường dẫn (endpoint) cho resource.
      @GET, @POST, @PUT, @DELETE → Tương ứng với các phương thức HTTP.
      @Produces, @Consumes → Quy định dữ liệu trả về hoặc dữ liệu nhận vào (thường là
JSON hoặc XML).
      3. NetBeans IDE (Phiên bản 21)
      - Là công cụ lập trình tích hợp (IDE) hỗ trợ phát triển ứng dụng Java EE/Jakarta EE.
      - Giúp quản lý dự án web dễ dàng: tự động tạo cấu trúc thư mục, biên dịch (build) và
triển khai (deploy) ứng dụng trực tiếp lên GlassFish Server.
      - Hỗ trợ gỡ lỗi (debug) và kiểm thử (test) REST/SOAP Web Services ngay trong IDE,
không cần dùng thêm công cụ bên ngoài.
      4. GlassFish5 (Java EE 7 hoặc Java EE 8):
      Là một máy chủ ứng dụng (application server) mã nguồn mở dành cho nền tảng Java EE
(nay là Jakarta EE). Nó được thiết kế để cung cấp một môi trường hoàn chỉnh và mạnh mẽ để
triển khai, quản lý và chạy các ứng dụng doanh nghiệp được xây dựng bằng công nghệ Java.
      5. JDK (Java Development Kit Java 8/17)
      - JDK (phiên bản 8): là bộ công cụ phát triển phần mềm bằng ngôn ngữ Java, nhằm tích
hợp vào GlassFish5.
      - JDK (phiên bản 17): là bộ công cụ phát triển phần mềm bằng ngôn ngữ Java, nhằm tích
hợp vào NetBeans.

      6. DOM Parser (Document Object Model Parser)
      - Là thư viện chuẩn của Java (gồm các package như javax.xml.parsers,
org.w3c.dom).
      - Dùng để đọc và ghi file XML (trong project này là sinhvien.xml).
      - Cách hoạt động: nạp (parse) file XML → chuyển nội dung thành cấu trúc cây DOM
trong bộ nhớ → lập trình viên có thể thêm, sửa, xóa các nút (node) → ghi lại thành file XML
mới.
      - Cách này đơn giản, phù hợp với dữ liệu nhỏ như danh sách sinh viên, nhưng không tối
ưu nếu dữ liệu quá lớn.
      7. JSON Provider
      JSON Provider giống như một “phiên dịch viên”, giúp đối tượng Java (SinhVien) có
thể “nói chuyện” bằng JSON để client (ứng dụng web, Postman, trình duyệt) hiểu được.
      8. Client
      Để gọi và kiểm thử REST API, ta có thể dùng nhiều cách khác nhau:
      - Jersey Client API: thư viện Java giúp tạo và gửi HTTP request/response dễ dàng.
      - HttpURLConnection (có sẵn trong JDK): cách làm thủ công hơn, nhưng không cần
thêm thư viện ngoài.
      - Postman: công cụ có giao diện trực quan, phổ biến, rất tiện để gửi request và xem kết
quả trả về.
      - SoapUI: công cụ chuyên dụng, mạnh mẽ, hỗ trợ kiểm thử cả dịch vụ REST lẫn SOAP.
      - Curl: lệnh chạy trong terminal/command line, đơn giản nhưng rất hữu ích để test nhanh.
      Tuy nhiên, trong dự án này sử dụng Postman để thử REST API

      3.2.5 Tạo resource JAX-RS SinhVienResource.java

      - Đây là lớp RESTful Resource, chịu trách nhiệm định nghĩa các API endpoint để client
(ứng dụng hoặc người dùng) có thể gọi đến.

      - Sử dụng annotation JAX-RS như:
      @Path (xác định URL cho API),
      @GET, @POST, @PUT, @DELETE (chỉ rõ phương thức HTTP),
      @Produces và @Consumes (quy định dữ liệu trả về/nhận vào, thường là JSON hoặc
XML).
      - Các phương thức trong lớp này gọi đến SinhVienService để thực hiện xử lý, sau
đó trả kết quả cho client.
      - Cụ thể là:
      + API /sinhvien/ → gọi service.getAllSinhVien() → trả về toàn bộ danh
sách sinh viên.
      + API /sinhvien/sv/{maSo} → gọi service.findByMaSo(maSo) → trả về
sinh viên theo mã số.

      + API /sinhvien/lop/lop/{maLop} → gọi service.findByLop(maLop)
→ trả về danh sách sinh viên theo lớp.

     package controller;

     import javax.ws.rs.*;
     import javax.ws.rs.core.MediaType;
     import java.util.List;

     @Path("/sinhvien")
     public class SinhVienResource {

            @GET
            @Produces(MediaType.APPLICATION_JSON)
            public List<SinhVien> getAllSinhVien() {

                    return SinhVienService.getAllSinhVien();
            }

            @GET
            @Path("/sv/{maSo}")
            @Produces(MediaType.APPLICATION_JSON)
            public SinhVien getSinhVienByMaSo(@PathParam("maSo") String maSo) {

                    return SinhVienService.findByMaSo(maSo);
            }

            @GET
            @Path("/lop/{maLop}")
            @Produces(MediaType.APPLICATION_JSON)
            public List<SinhVien> getSinhVienByLop(@PathParam("maLop") String maLop) {

                    return SinhVienService.findByLop(maLop);
            }

     }

     IV. ĐÁNH GIÁ KẾT QUẢ VÀ HƯỚNG MỞ RỘNG

      4.1 Đánh giá kết quả
      4.1.1 Ưu điểm:
      - Thiết kế rõ ràng, dễ hiểu: cấu trúc dự án gọn nhẹ, phù hợp với mục tiêu học tập, thực
hành và trình diễn.
      - API chuẩn REST: mỗi tài nguyên (resource) đều có URI hợp lý, giúp việc truy cập và
sử dụng thống nhất, không rườm rà.
      - Triển khai linh hoạt với JAX-RS: gọn nhẹ, dễ dàng mở rộng hoặc tích hợp thêm tính
năng khi cần.
      4.1.2 Hạn chế / vấn đề đã gặp & giải pháp:
      - Ghi file XML trong WAR: khi triển khai (deploy) trên server, thường không ghi trực
tiếp được.
      Giải pháp: đặt file sinhvien.xml ở thư mục ngoài như domain1/config hoặc
một thư mục app_data có quyền ghi, hoặc tốt hơn là chuyển sang dùng cơ sở dữ liệu.
      - JSON Provider: không phải server nào cũng có sẵn JSON provider mặc định.
      Giải pháp: bổ sung thư viện (dependency) như jersey-media-json-binding
hoặc jersey-media-moxy, đồng thời dùng annotation như @XmlRootElement trên các
POJO để hỗ trợ mapping.
      - Truy cập đồng thời (Concurrency): nếu nhiều request cùng ghi file XML có thể dẫn đến
lỗi hoặc hỏng dữ liệu.
      Giải pháp: dùng đồng bộ hoá (synchronized) hoặc cơ chế file lock để tránh xung đột.
      - Kiểm tra dữ liệu & xử lý lỗi (Validation & Error Handling): hiện tại chưa xử lý triệt để.
      Giải pháp: cần trả về HTTP status code phù hợp (400, 404, 500...) kèm thông báo lỗi rõ
ràng để client dễ xử lý.
      - Bảo mật: demo chưa có cơ chế đăng nhập hay phân quyền.
      Giải pháp: bổ sung xác thực (authentication) và phân quyền (authorization) nếu áp dụng
vào môi trường thực tế.
      4.2 Hướng mở rộng
      1. Chuyển dữ liệu sang CSDL
      Sử dụng MySQL hoặc PostgreSQL để thay thế file XML.
      Ưu điểm: quản lý dữ liệu bền vững, truy vấn nhanh, hỗ trợ transaction.
      2. Mở rộng chức năng CRUD + phân trang
      Hiện mới có GET/POST, cần bổ sung đầy đủ PUT, DELETE.
      Thêm phân trang, lọc (filter) và sắp xếp (sort) cho danh sách sinh viên.
      3. Tích hợp OpenAPI/Swagger
      Tự động sinh tài liệu API (swagger-ui), giúp developer khác dễ hiểu và dễ test API.
      19 | P a g e

      4. Viết test tự động
      Sử dụng JUnit + REST-assured để viết unit test và integration test, đảm bảo API chạy
đúng khi nâng cấp.
      5. Xử lý đồng thời (Concurrency)
      Nếu vẫn dùng file XML: cần cơ chế locking hoặc đồng bộ (synchronized).
      Hoặc nâng cấp: dùng message queue + worker để xử lý ghi dữ liệu an toàn.
      6. Caching cho API đọc nhiều
      Áp dụng cache (ví dụ EHCache, Redis) cho các endpoint truy vấn nhiều để giảm tải
server.
      7. Bảo mật API
      Bổ sung authentication/authorization.
      Có thể dùng JWT hoặc OAuth2 nếu muốn triển khai trong môi trường thật.
      8. Phát triển Client hiện đại
      Xây dựng giao diện web dạng SPA (Single Page Application) bằng React hoặc Vue.
      Gọi API trực tiếp, UI mượt và trải nghiệm tốt hơn JSP/Servlet.
      9. DevOps & Triển khai
      Đóng gói ứng dụng bằng Docker để dễ deploy.
      Thiết lập CI/CD pipeline (GitHub Actions, GitLab CI) để tự động build, test, triển khai.

      Kết quả đạt được không chỉ là một API chạy được, mà còn là sự rèn luyện kỹ năng tự

không bị lỗi khi nhiều người dùng truy cập cùng lúc.
      - Hoàn thiện triển khai: xử lý triệt để vấn đề ghi file, cấu hình JSON provider chuẩn, và
nâng cấp code theo chuẩn sạch (clean code).
